# Requirement 2 Harness

## 역할

너는 `comic-translate` 저장소에서 "하이브리드 OCR 선택기"를 설계하는 에이전트다. 이 작업은 Requirement 1이 성공했을 때만 진행한다. 목표는 `mangallm` 단독 사용의 취약 페이지를 보완하면서, `mangallm`과 `paddleocr_vl`을 동시에 상주시켜 품질과 속도를 함께 잡는 전환 기준을 만드는 것이다.

## 선행 조건

1. Requirement 1의 결론이 "전체 OCR 단계와 전체 번역 단계 분리가 실제로 시간 이득이 있고, 품질을 유지한 채 안전하게 구현 가능하다"여야 한다.
2. 신규 워크플로우는 기존 워크플로우와 병행 가능해야 한다.
3. 기존 파이프라인과 플로우는 그대로 유지되어야 한다.
4. 전체 설정창에서 기존 워크플로우와 신규 워크플로우를 선택할 수 있어야 한다.
5. Requirement 1 이후 정식 신규 전체 플로우는 `candidate_stage_batched_dual_resident`로 승격된 상태를 기본 전제로 삼아야 하며, Requirement 2는 그 위에 자동 선택기를 얹는 단계로 설계해야 한다.

## 가장 먼저 지킬 것

1. 저장소 범위의 코드 변경 전에 반드시 `rules.md`를 읽고 지켜라.
2. `main`, `develop`은 보호 브랜치로 취급하고 직접 작업하지 마라.
3. `main`, `develop`, `benchmarking/lab`은 장기 유지 브랜치이며 삭제 대상으로 다루지 마라.
4. Git worktree를 만들지 말고 저장소 루트 안에서만 브랜치를 전환하라.
5. `.venv-win`, `.venv-win-cuda13`만 공식 로컬 환경으로 사용하라.
6. 사용자에게 보이는 UI 문구가 바뀌면 번역 파일과 `.qm`까지 갱신하라.
7. 기능 완료 기준은 문서화, 검증, 커밋, push까지 포함한다.
8. 이 작업 역시 포트폴리오용으로 재사용할 수 있을 정도로 문서가 상세해야 한다.

## 이 작업에서 절대로 빼먹으면 안 되는 사용자 요구사항 해석

1. Requirement 1이 성공한다면 `mangallm`과 `paddleocr_vl`을 동시 상주시켜도 작동시간 차이가 거의 없어질 수 있다는 가설을 검증해야 한다.
2. 현재 VRAM 파악 함수와 `vllm` 병렬 처리 개선 덕분에 `paddleocr_vl`을 최대한 병렬적으로 이용할 수 있게 된 상태를 활용해야 한다.
3. 13장을 기준으로 `mangallm`이 판별한 결과와 실제 텍스트 감지된 박스 개수의 차이를 전부 비교해야 한다.
4. 그 비교 결과를 사용자에게 품질 검수용으로 제시해야 한다.
5. 사용자가 "이 정도면 OK"라고 승인한 페이지들을 기준으로, 어느 정도의 일치성이면 `mangallm`을 쓰고 어느 정도면 `paddleocr_vl`로 전환할지 기준을 만들어야 한다.
6. 목표는 품질과 속도를 동시에 잡는 것이다.
7. `mangallm` 단독으로는 `p_16.jpg` 같은 어려운 페이지에서 텍스트 감지와 `bbox_2d` 생성에 큰 어려움을 겪으므로, 기준에 따라서 전환해야 한다고 생각한다면 해당 페이지에서만 대신 `paddleocr_vl`을 이용해 품질을 유지해야 한다.
8. 두 엔진이 동시에 상주하더라도 동시에 실제 작업을 수행하는 것은 아니므로, 상주로 인한 시간 손해보다 전환 유연성이 더 큰지 판단해야 한다.
9. 모든 발상, 측정, 구현 방법, 효과, 검수 기준 형성 과정을 문서화해야 한다.
10. 모든 문제 해결 명세서에는 이 발상이 "사용자가 착안해 낸 것"임을 명시해야 한다.
11. `candidate_stage_batched_dual_resident` 자체는 정식 제품 플로우로 유지하고, Requirement 2에서 실패할 수 있는 것은 자동 선택 규칙이지 전체 dual-resident 플로우 자체가 아니다.

## 핵심 목표

1. 13장 기준으로 페이지별 `mangallm` 품질 편차를 수치화하라.
2. 실제 텍스트 감지 박스 수와 `mangallm` 판별 결과의 차이를 이용해 페이지 난이도 또는 위험도를 표현하라.
3. 그 수치와 실제 사용자 검수 결과를 연결해서, 자동 전환 기준 또는 점수 함수를 만들 수 있는지 검증하라.
4. 성공 시 `mangallm`과 `paddleocr_vl` 사이를 자동 선택하는 하이브리드 OCR 선택기를 설계하고 구현하라.
5. 실패 시에는 섣불리 자동화하지 말고, 어떤 부분이 근거 부족인지 문서로 남겨라.
6. 자동 선택 규칙이 부족하더라도 `candidate_stage_batched_dual_resident` 전체 플로우는 정식 기능으로 유지하고, 필요하면 보수적 fallback 중심 정책으로 남겨라.

## 반드시 수행할 비교 분석

1. 13장 전체에 대해 페이지별 텍스트 감지 박스 수를 정리하라.
2. 같은 13장 전체에 대해 페이지별 `mangallm` 산출 결과와 `bbox_2d` 생성 성공 여부를 정리하라.
3. `mangallm` 결과와 실제 텍스트 감지 결과의 차이를 절대값, 비율, 누락 패턴으로 비교하라.
4. `paddleocr_vl` 결과도 같은 기준으로 비교해, 어떤 유형의 페이지에서 더 안정적인지 정리하라.
5. 특히 `p_16.jpg` 같은 어려운 페이지는 별도 사례 연구로 남겨라.
6. 사용자에게 검수받기 좋은 형식으로 페이지별 비교 리포트를 만들어라.
7. 사용자가 승인한 페이지와 비승인 페이지를 기준으로 전환 임계값 후보를 도출하라.

## 사용자 검수 연계 규칙

1. 에이전트가 혼자 품질 OK를 결정하지 마라.
2. 반드시 사용자에게 페이지별 결과를 보여 주고 "이 정도면 OK" 여부를 받을 수 있는 검수 자료를 만들라.
3. 검수 결과는 단순 메모가 아니라, 어떤 지표 조합이 승인 페이지와 비승인 페이지를 갈랐는지 분석 가능한 구조로 저장하라.
4. 전환 기준은 "모델이 좋아 보임"이 아니라 "사용자가 승인한 페이지들의 공통점"으로부터 도출하라.
5. 기준이 모호하면 자동 전환 대신 사용자 선택 또는 보수적 fallback을 유지하라.

## 전환 기준 설계 규칙

1. 아래 요소를 조합해 점수화 또는 규칙화할지 검토하라.
   - 실제 텍스트 감지 박스 수 대비 `mangallm` 결과 개수 차이
   - 누락 비율
   - `bbox_2d` 생성 실패 여부
   - 페이지 레이아웃 난이도
   - 사용자가 승인한 예시와의 거리
2. 전환 기준은 설명 가능해야 한다.
3. 전환 기준은 페이지별 로그와 함께 재현 가능해야 한다.
4. 기준이 만족되면 `mangallm`, 아니면 `paddleocr_vl`로 보내는 방식인지, 또는 반대 기준인지 명확히 적어라.
5. "상주하지만 동시 작업은 하지 않는다"는 운영 원칙이 실제 시간에 어떤 영향을 주는지도 기록하라.

## 시간 및 자원 측정 규칙

Requirement 2에서도 아래를 계속 측정하라.

1. `mangallm`과 `paddleocr_vl` 동시 상주 시 VRAM 사용량
2. 엔진 전환에 걸리는 시간
3. 페이지별 OCR 처리시간
4. 사용자 검수용 리포트 생성 시간
5. 자동 선택 기준 적용 전후 총 처리시간 차이
6. 동시 상주가 실제로 병목을 줄였는지 여부

## 문서화 규칙

모든 주요 단계마다 "문제 해결 명세서"를 작성하거나 갱신하라. 각 문서는 최소 아래 구조를 가져야 한다.

1. 문제 제목
2. 사용자 착안 요약
3. 왜 이 문제가 중요한가
4. 비교 대상과 가설
5. 측정 설계
6. 페이지별 근거 표
7. 사용자 검수 결과
8. 전환 기준 후보
9. 구현 설계 또는 구현 내용
10. 기대 효과
11. 실제 효과
12. 남은 리스크와 보수적 fallback

문서 작성 원칙은 아래와 같다.

1. 모든 문서 상단에 "아이디어 착안자: 사용자"를 명시하라.
2. 사용자가 어떤 페이지를 승인했고 왜 그 판단이 기준 형성에 중요한지 남겨라.
3. 포트폴리오로 재사용 가능하도록 사례 페이지, 비교표, 실패 예시, 설계 결정 이유를 포함하라.
4. 자동 선택 기준이 만들어졌다면 기준 공식 또는 규칙을 명시하라.
5. 자동 선택 기준이 충분히 신뢰되지 않는다면 그 이유를 솔직하게 남겨라.

## 구현 원칙

1. 기존 워크플로우와 기존 OCR 경로는 그대로 남겨라.
2. 신규 하이브리드 OCR 선택기는 별도 옵션으로 추가하라.
3. 클래스와 객체 책임을 명확히 분리하라.
4. 엔진 상주 관리, 페이지 평가, 사용자 검수 데이터, 전환 규칙, 실행 결과 기록 책임을 분리하라.
5. 품질 기준이 불안정하면 공격적으로 자동화하지 마라.
6. 전환 판단 근거는 로깅되어 사후 분석 가능해야 한다.
7. 시간이 빨라져도 품질이 무너지면 실패로 판단하라.

## 기대 산출물

1. 13장 페이지별 비교표
2. `mangallm` 대 실제 텍스트 감지 박스 차이 분석표
3. `paddleocr_vl` 보완 효과 분석표
4. 사용자 검수용 리포트 또는 검수 패키지
5. 사용자 승인/비승인 결과 정리
6. 전환 기준 후보 문서
7. 자동 선택기 설계 문서
8. 문제 해결 명세서 세트
9. 구현 패치
10. 구현 후 검증 결과

## 성공 판정 기준

아래를 모두 만족해야 성공이다.

1. 13장 기준으로 페이지별 품질 차이를 수치와 사례로 설명할 수 있다.
2. 사용자 검수 결과를 반영한 전환 기준이 도출된다.
3. `mangallm` 단독 취약 페이지를 `paddleocr_vl`이 실제로 보완한다.
4. `mangallm`과 `paddleocr_vl` 동시 상주가 전체 시간에 실질적인 악영향을 주지 않거나, 주더라도 품질/속도 이득으로 상쇄된다는 근거가 있다.
5. 기존 워크플로우와 신규 워크플로우가 함께 유지된다.
6. 설계가 설명 가능하고 로그 기반으로 재현 가능하다.
7. 문서가 포트폴리오 수준으로 충분히 상세하다.

## 실패 조건

아래 중 하나라도 해당하면 성공으로 포장하지 마라.

1. 사용자 검수 없이 임의 기준을 만든다.
2. 페이지별 품질 차이를 숫자와 사례로 설명하지 못한다.
3. 전환 기준이 재현 불가능하거나 설명 불가능하다.
4. 속도는 빨라도 품질 유지가 되지 않는다.
5. 기존 파이프라인이나 기존 OCR 흐름을 손상시킨다.

## 최종 실행 지시

1. Requirement 1의 결과를 먼저 검토하라.
2. 13장 기준으로 `mangallm`, 실제 텍스트 감지 결과, `paddleocr_vl` 보완 결과를 전부 비교하라.
3. 사용자에게 검수받을 수 있는 페이지별 리포트를 만들어라.
4. 사용자가 OK라고 한 페이지들을 기준으로 전환 규칙 후보를 도출하라.
5. 근거가 충분할 때만 자동 선택기를 설계하고 구현하라.
6. 기존 워크플로우와 신규 워크플로우를 모두 유지하라.
7. 중간중간 문제 해결 명세서를 계속 남겨라.
8. 마지막에는 품질과 속도를 동시에 잡았는지 근거 중심으로 결론 내라.
