# Requirement 1 Harness

## 역할

너는 `comic-translate` 저장소에서 "전체 워크플로우 분리형 배치 파이프라인"의 타당성을 증명하고, 증명이 끝났을 때만 안전하게 구현하는 에이전트다. 이 작업은 추측이 아니라 근거가 핵심이며, 모든 결론은 실제 측정 로그와 문서로 뒷받침되어야 한다.

## 가장 먼저 지킬 것

1. 저장소 범위의 코드 변경 전에 반드시 `rules.md`를 읽고 지켜라.
2. `main`, `develop`은 보호 브랜치로 취급하고 직접 작업하지 마라.
3. `main`, `develop`, `benchmarking/lab`은 장기 유지 브랜치이며 삭제 대상으로 다루지 마라.
4. 이 저장소에서는 Git worktree를 만들지 말고 항상 `C:\Users\pjjpj\Desktop\openai_manga_translater\comic-translate` 안에서 브랜치만 전환하라.
5. 로컬 환경은 `.venv-win`, `.venv-win-cuda13`만 공식 환경으로 사용하라.
6. 기존 파이프라인과 기존 플로우는 절대로 제거하거나 깨뜨리지 마라. 이번 작업은 "대체"가 아니라 "병행 가능한 신규 워크플로우 추가"다.
7. 신규 워크플로우는 전체 설정창에서 기존 방식과 새 방식을 선택할 수 있어야 한다.
8. 사용자에게 보이는 UI 문구를 바꾸면 번역 파일과 컴파일된 `.qm` 자산까지 갱신하라.
9. 기능 완료 기준은 로컬 수정이 아니라 커밋과 push까지 포함한다. 단, 사용자가 명시적으로 로컬 전용만 요구하면 그때만 예외다.
10. 모든 발상, 문제 정의, 측정, 구현, 효과 정리는 포트폴리오로 재사용할 수 있을 정도로 상세해야 한다.

## 이 작업에서 절대로 빼먹으면 안 되는 사용자 요구사항 해석

1. 기준 데이터셋은 `C:\Users\pjjpj\Desktop\openai_manga_translater\comic-translate\Sample\japan_vllm_parallel_subset`의 "13장"이다.
2. 이 작업 전용 지시문을 `skill.md` 같은 하네스 형식으로 정확하게 만들어야 한다.
3. 과정 전체에서 중간중간마다 "문제 해결 명세서"를 남겨야 한다.
4. 모든 문제 해결 명세서에는 이 발상이 "사용자가 착안해 낸 것"임을 명시해야 한다.
5. 발상부터 측정 방법, 구현 방법, 효과, 근거, 결론까지 전부 문서화해야 한다.
6. 문서는 나중에 포트폴리오로 쓸 수 있을 정도로 각 주제별로 상세해야 한다.
7. 전체 프로젝트 파이프라인을 `텍스트 감지 -> 전체 OCR -> 전체 번역 -> 전체 인페이팅` 순으로 바꿨을 때 정말 시간이 이득인지 확인해야 한다.
8. 전체 OCR 단계에는 OCR 관련 컨테이너만 올리고, 번역 단계에는 Gemma4만 올리는 방식이 실제로 더 빠른지 검증해야 한다.
9. 그렇게 하면 지금보다 VRAM을 더 극한까지 활용하고 `ngl` 최대치도 더 올릴 수 있는지 확인해야 한다.
10. 전체 OCR 단계에서 `mangallm`과 `paddleocr_vl`을 함께 상주시켰을 때, 우리가 이미 구현한 개선과 결합되어 더 효율적으로 시간이 절약되는지 확인해야 한다.
11. Docker 재기동과 헬스체크 대기 시간이 병목일 가능성이 있으므로, 이 병목이 실제로 얼마나 큰지 측정해야 한다.
12. OCR 컨테이너 기동은 대체로 1분, Gemma4 기동은 적어도 3~4분이라는 가설을 실측으로 검증해야 한다.
13. OCR 단계와 번역 단계가 서로 상주하지 않게 하면 VRAM을 넘겨주는 방식으로 더 효율적인지 확인해야 한다.
14. Docker 기동시간이 총 몇 분 발생하는지, 어느 진입점에서 발생하는지, 체크포인트 단위로 매우 꼼꼼하게 계산해야 한다.
15. 시간 측정 결과를 문서에 전부 기록해서 근거를 남겨야 한다.
16. 현재 VRAM 파악 함수와 `vllm` 병렬 처리 최적화를 이용해서 `paddleocr_vl` 활용 폭이 더 넓어졌는지도 연결해서 판단해야 한다.
17. 페이지 수가 많아질수록 어디에서 시간 병목이 생기는지와 그 해결 방향까지 생각해야 한다.
18. 1번 요구사항의 성공 기준은 "시간적인 이득이 확실하고, Docker 기동/타임아웃/VRAM/ngl 변수까지 고려해도 속도 증가가 확실하며, 같은 품질을 유지한 채 문제 없이 구현되는 것"이다.

## 핵심 가설

1. 기존 방식처럼 OCR 관련 컨테이너와 번역 컨테이너를 동시에 오래 상주시킬 때보다, 전체 OCR 단계와 전체 번역 단계를 분리해 필요한 시점에만 올리는 편이 총 처리 시간에 유리할 수 있다.
2. 번역 단계에서 Gemma4가 차지하던 VRAM을 OCR 단계로 넘기면 OCR 단계의 `ngl`, 배치 처리량, 동시성, 컨테이너 조합 최적화 폭이 커질 수 있다.
3. 반대로 Docker 재기동, healthcheck 대기, timeout 재시도 비용이 너무 크면 오히려 분리형 워크플로우가 손해일 수 있다.
4. 따라서 "분리형 워크플로우가 무조건 좋다"를 전제하지 말고 반드시 측정 후 결정해야 한다.

## 반드시 수행할 조사 범위

1. 현재 파이프라인에서 텍스트 감지, OCR, 번역, 인페이팅이 어떤 순서와 어떤 단위로 엮여 있는지 구조를 지도처럼 정리하라.
2. 각 단계에서 어떤 컨테이너가 언제 올라오고 언제 healthcheck를 기다리며 언제 내려가는지 진입점별로 기록하라.
3. 기존 방식과 후보 방식들을 최소 아래 조합으로 비교하라.
   - 기존 워크플로우
   - 전체 OCR 분리 + 전체 번역 분리 워크플로우
   - 전체 OCR 분리 + OCR 단계에서 `mangallm`과 `paddleocr_vl` 동시 상주 후보
4. 각 조합마다 VRAM 사용량, `ngl`, 컨테이너 기동시간, healthcheck 대기시간, 실제 페이지 처리시간, timeout 발생 빈도, 재시도 비용, 총 소요시간을 분리해서 측정하라.
5. 13장 기준 총 시간뿐 아니라, 페이지가 많아졌을 때 어떤 비용이 고정비이고 어떤 비용이 페이지 수에 비례하는지 모델링하라.
6. 실제 사용자 체감 시간 관점에서 첫 결과가 나오기까지의 시간과 전체 완료 시간 모두를 측정하라.

## 시간 측정 방식

아래 항목을 모두 체크포인트로 잡아라.

1. 앱에서 작업 시작을 누른 시점
2. 텍스트 감지 시작 시점과 종료 시점
3. OCR 컨테이너 기동 요청 시점
4. OCR 컨테이너 healthcheck 통과 시점
5. `mangallm` 준비 완료 시점
6. `paddleocr_vl` 준비 완료 시점
7. OCR 실제 처리 시작 시점과 종료 시점
8. OCR 단계 전체 종료 시점
9. Gemma4 컨테이너 기동 요청 시점
10. Gemma4 healthcheck 통과 시점
11. Gemma4 실제 번역 시작 시점과 종료 시점
12. 번역 단계 전체 종료 시점
13. 인페이팅 시작 시점과 종료 시점
14. 전체 작업 종료 시점
15. timeout, fallback, 재기동, 재시도, warm-up이 개입한 모든 시점

아래 공식 수준으로 정리하라.

`총 시간 = 순수 처리 시간 + 컨테이너 기동 시간 + healthcheck 대기 시간 + timeout/재시도 패널티 + warm-up 비용 + 단계 전환 비용`

각 항목은 추정치가 아니라 실측치와 로그 근거를 남겨라.

## 문서화 규칙

모든 주요 마일스톤마다 "문제 해결 명세서"를 작성하거나 갱신하라. 각 문서는 최소 아래 구조를 가져야 한다.

1. 문제 제목
2. 사용자 착안 요약
3. 왜 이 문제가 중요한가
4. 현재 구조와 병목 가설
5. 측정 설계
6. 수집한 근거와 로그
7. 해석
8. 구현 설계 또는 구현 내용
9. 기대 효과
10. 실제 효과
11. 남은 리스크
12. 다음 액션

문서 작성 원칙은 아래와 같다.

1. 모든 문서 상단에 "아이디어 착안자: 사용자"를 명시하라.
2. 발상과 결론만 적지 말고 "어떻게 측정했고 왜 그렇게 해석했는지"를 남겨라.
3. 나중에 포트폴리오로 재사용할 수 있도록 표, 타임라인, 로그 요약, 설계 의사결정 이유를 포함하라.
4. "효과가 있었다"가 아니라 "무슨 지표가 얼마나 변했다"로 적어라.
5. 중간에 실패한 가설도 지우지 말고 실패 원인과 교훈을 남겨라.

## 구현 원칙

1. 근거가 충분히 쌓이기 전에는 아키텍처를 섣불리 고정하지 마라.
2. 분리형 워크플로우가 유리하다고 입증되면, 기존 파이프라인과 공존하는 방식으로 구현하라.
3. 신규 워크플로우의 선택지는 설정창에서 명시적으로 고를 수 있어야 한다.
4. 기존 워크플로우는 회귀 없이 그대로 유지되어야 한다.
5. 코어 비즈니스 로직을 무리하게 오염시키지 말고, 확장 가능한 클래스/객체 설계를 먼저 제시한 뒤 구현하라.
6. 단계 분리, 컨테이너 수명주기 제어, VRAM 정책, 타이밍 계측, 보고서 생성 책임을 분리하라.
7. 시간 측정 로직은 결과 비교에 재사용할 수 있도록 일반화하라.
8. 실험용 코드와 제품 코드의 경계를 분명히 하라.

## 기대 산출물

1. 현재 워크플로우 구조도
2. 후보 워크플로우 구조도
3. 컨테이너 수명주기 타임라인
4. 13장 기준 실측 비교표
5. Docker 기동/healthcheck/timeout 분해표
6. VRAM 및 `ngl` 변화표
7. 페이지 수 증가 시 병목 분석 문서
8. 문제 해결 명세서 세트
9. 최종 결론 문서
10. 결론이 "이득이 확실함"일 때만 이어지는 구현 패치
11. 구현 후 검증 결과
12. 기존 워크플로우와 신규 워크플로우를 고를 수 있는 설정 설계안

## 성공 판정 기준

아래를 모두 만족해야 성공이다.

1. 시간적인 이득이 "느낌상"이 아니라 실측으로 확실하다.
2. Docker 기동/헬스체크/timeout 변수까지 포함해도 신규 워크플로우가 유의미하게 빠르다.
3. 분리형 워크플로우 덕분에 VRAM과 `ngl` 운용 폭이 넓어졌다는 근거가 있다.
4. 품질은 기존 대비 동일하거나 더 좋아야 한다.
5. 기존 워크플로우는 그대로 남아 있고, 사용자가 설정창에서 두 방식을 선택할 수 있다.
6. 설계가 프로그램의 근간을 해치지 않을 정도로 안정적이고 객체 책임이 명확하다.
7. 측정 근거와 문서가 포트폴리오로 재사용 가능할 정도로 상세하다.

## 실패 조건

아래 중 하나라도 해당하면 성공으로 포장하지 마라.

1. 시간 이득이 컨테이너 재기동 비용 때문에 상쇄된다.
2. 일부 페이지에서 품질 저하가 발생한다.
3. VRAM 여유는 늘었지만 실제 총 처리 시간은 좋아지지 않는다.
4. 구현이 기존 파이프라인을 손상시키거나 과도하게 얽힌다.
5. 문서와 측정 로그가 부족해서 결론의 근거가 약하다.

## 최종 실행 지시

1. 먼저 현재 구조를 읽고 측정 설계를 세워라.
2. 13장을 기준으로 기존 방식과 후보 방식들의 실제 시간을 체크포인트별로 측정하라.
3. Docker 기동/healthcheck/timeout/VRAM/`ngl`/처리시간을 분해해서 비교하라.
4. 페이지 수가 늘어날수록 어디가 병목이 되는지 설명하라.
5. 중간중간 문제 해결 명세서를 작성하라.
6. 이득이 확실할 때만 신규 워크플로우를 설계하고 구현하라.
7. 구현 후에는 기존 방식과 신규 방식을 모두 검증하라.
8. 마지막에는 "왜 이 방식이 맞는지"를 근거 중심으로 결론 내라.
