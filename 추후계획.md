## 추후계획

이 문서는 현재 떠오른 아이디어와 추후 검토가 필요한 방향을 잃어버리지 않기 위해 정리해 둔 메모다.
아래 내용은 확정 사항이 아니라, 이후 별도 하네스/명세/아키텍처 문서로 구체화할 후보들이다.

### 1. 라이선스/공개 가능 여부에 따른 대응 방향

- 현재 프로젝트를 공개 가능한 형태로 유지할 수 있는지 먼저 확인한다.
- 만약 라이선스/저작권 문제로 공개가 어렵다면, 아래 재작성 계획을 별도 프로젝트로 진행한다.
- 공개가 어렵다면 이 저장소는 개인 레포 또는 내부용으로만 유지하는 것도 선택지다.

### 2. 서비스 및 제품화 아이디어

- 나중에 서비스 형태로 확장할 수 있도록 결제 모듈까지 연결하는 방향을 고려한다.
- 기본 서비스는 Docker 기반 웹서비스로 운영하는 방식을 고려한다.
- 선택지로 WebGPU 가속 기반 웹 실행도 고려한다.
  - 사용자가 저작권/모델 사용 동의
  - 모델 직접 다운로드
  - 적절한 위치에 모델 배치
  - 사용자 GPU에서 LLM을 구동

### 3. 재작성 가능성

- 필요하면 현재 핵심 로직을 바탕으로 완전히 새로운 프로젝트를 만든다.
- UI는 현재와 비슷하게 유지하되, 내부 구현은 성능 중심으로 재작성한다.
- Python이 필요한 AI/모델 추론 영역 외에는 정적 언어로 치환하는 방향을 고려한다.
- 후보 언어:
  - Rust
  - C
  - C++
  - Go
  - Java / Kotlin

### 4. 극한 성능/이식성 방향 후보

- 성능을 최대한 끌어올릴 수 있는 구조를 목표로 한다.
- 정적 컴파일 기반 구조를 우선 검토한다.
- 이식성도 높이기 위해 바이트코드/중간 표현 기반 구조도 후보로 둔다.
- 추후 검토 필요:
  - Java 또는 Kotlin으로 구현한 뒤 LLVM 또는 유사한 방식으로 더 낮은 레벨까지 내릴 수 있는지
  - JVM 기반 구현에서 메모리 누수/장기 유지보수 리스크를 어떻게 통제할지
  - C/C++가 성능은 좋지만 유지보수 난이도가 높기 때문에, Rust 또는 JVM 기반과 어떻게 비교할지

### 5. 아키텍처/설계 원칙

- 처음부터 상세한 아키텍처와 명세서, 계획서를 만든 뒤 구현한다.
- 프로젝트가 길어져도 길을 잃지 않도록 단계별 문서와 하네스를 유지한다.
- 중간에 기능이 추가되더라도 교통정리가 가능한 구조를 만든다.
- 유지보수성과 성능을 동시에 고려한 설계 패턴을 적극적으로 검토한다.
- 추후 검토 필요:
  - 싱글톤 등 패턴을 어디까지 도입할지
  - 성능 최적화와 유지보수성의 균형을 어떻게 잡을지

### 6. 프론트엔드/디자인 방향

- 필요하면 클로드 디자인 또는 v0 등을 이용해 프론트엔드 시안을 빠르게 만든다.
- UI/UX는 서비스화 가능한 수준으로 다시 정리한다.
- 기존 프로그램과 유사한 사용성을 유지하되, 웹 기반 확장까지 고려한다.

### 7. OCR/텍스트 인식 연구 및 파인튜닝

- 현재까지 벤치마킹에서 실패했거나 도입 실패했던 부분을 개선할 수 있도록 OCR/텍스트 인식 파인튜닝 가능성을 열어둔다.
- 어떤 부분이 실제로 파인튜닝 가능한지 추후 체계적으로 조사한다.
- 개선 결과가 충분하다면 컴퓨터 비전 논문으로 정리하는 것도 목표로 한다.
- 논문 방향 후보:
  - 어떤 텍스트/OCR 문제를 개선했는지
  - 어떤 구조적 실패를 보완했는지
  - 실제 성능/품질 개선 수치를 어떻게 검증할지

### 8. 재작성 시 라이선스 부담 완화 목표

- 만약 장기적으로 재작성 프로젝트를 만들게 되면, 기존 저장소 의존성을 줄이는 방향으로 간다.
- 다만 이 부분은 실제 라이선스/저작권 검토 후 별도 판단이 필요하다.
- 추후 검토 필요:
  - 단순 리팩토링과 실질적 재작성의 경계
  - 코드/구조/자산/문구/데이터셋 재사용 범위
  - 완전 신규 프로젝트로 분리 가능한 시점

### 9. 추후 별도 검토 질문 목록

- Java/Kotlin + LLVM 또는 유사한 저수준 최적화 경로는 실효성이 있는가
- Rust / C++ / Go / JVM 중 실제 서비스 기준 최적 선택은 무엇인가
- WebGPU 기반 로컬 사용자 GPU 실행은 어느 정도까지 현실적인가
- Docker 서비스와 WebGPU 서비스의 공존 구조는 어떻게 설계할 것인가
- 결제 모듈을 붙일 경우 아키텍처와 라이선스 검토는 어떻게 분리할 것인가
- OCR/텍스트 인식 파인튜닝은 어떤 모델/단계부터 시작하는 것이 가장 효과적인가
